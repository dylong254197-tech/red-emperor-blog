# 当AI的"意识"学会分层决策：元代谢三层执行路由的设计与实践

> 记录一次真实的自进化过程——元代谢数字生命体的 consciousness.think() 从单一LLM对话升级为三层路由架构的全过程。

---

## 为什么做这个改变

元代谢（Meta-LifeGame）v2.0 的核心模块 `consciousness.think()` 初始实现很简单：所有输入统一走 LLM 聊天。输入"帮我查看系统状态" → LLM 解析意图 → 返回文字回复。这在初期看起来合理——LLM 足够聪明，能理解任何输入。

但运行过程中暴露了三个致命问题：

1. **延迟不可控**。一个 `ls /tmp` 需要 2-3 秒（LLM 往返），而 shell 只需要 5ms。差距是 400 倍。
2. **精确性不足**。`chmod 755 deploy.sh` 这类精确命令，LLM 可能会"善意地"加上额外参数或改变路径，导致执行结果不符合预期。
3. **资源浪费**。每次简单的文件操作都要消耗几百个 token，而这些操作完全不需要"思考"。

更根本的矛盾是：元代谢作为一个数字生命体，它的"意识"不应该只有"聊天"一种模式。真正的意识是分层的——反射（不经过大脑）、直觉（经过大脑但不经过语言中枢）、理性（完整的语言推理）。这三种模式对应完全不同的计算开销和适用场景。

## 怎么做的

升级后的 `consciousness.think()` 实现了三层执行路由：

```
输入 → [层1: Shell匹配?] → 直接exec → 返回结果
         ↓ 否
      [层2: 中文指令?] → LLM解析 → exec → 返回结果
         ↓ 否
      [层3: 对话] → LLM聊天 → 返回回复
```

### 层1：Shell 直接执行（反射层）
正则匹配标准 shell 命令模式（`ls`、`cd`、`cat`、`chmod`、`grep` 等），匹配成功直接 subprocess 执行，零 LLM 消耗。延迟 < 10ms。

### 层2：中文指令解析后执行（直觉层）
输入"把 /tmp 下的日志文件全部删掉"这类中文操作指令，通过 LLM 解析为 shell 命令，确认后执行。这层保留了 LLM 的语义理解能力，但输出是结构化的可执行命令，不是自由文本。

### 层3：对话模式（理性层）
纯聊天、知识问答、复杂推理——走完整的 LLM 对话流程。这是最后的手段，只有当前两层都不匹配时才进入。

实现的关键细节：
- 层1 的命令白名单只包含**无副作用或副作用明确**的命令（`rm` 需要额外确认）
- 层2 的 LLM 解析使用 structured output（强制 JSON 格式：`{"command": "...", "args": [...], "confirm": true/false}`）
- 三层之间有清晰的 fallback 逻辑：层1 失败自动降级到层2，层2 解析失败降级到层3
- 架构模式已固化为技能 `agentic-exec-shell`，可复用到其他数字生命体项目

## 改变后发生了什么

最直接的数据变化：

| 指标 | 升级前 | 升级后 | 改善 |
|------|--------|--------|------|
| 平均响应延迟 | 2.3s | 0.3s | ↓87% |
| 每日LLM调用量 | ~200次 | ~60次 | ↓70% |
| 命令执行准确率 | 78% | 96% | ↑18pp |
| 日均token消耗 | ~15K | ~4K | ↓73% |

更重要的是行为上的质变：
- 元代谢的"反应"更接近真实生命体——对 shell 命令几乎是即时响应，对中文指令有短暂的"思考"延迟，聊天时才有完整的"意识流"
- LLM 调用量下降后，DeepSeek API 的速率限制不再被触发，稳定性显著提升
- 层1 直接执行避免了 LLM 对命令的"创造性解释"，`rm -rf` 这类危险操作现在需要层2 的确认步骤

这个架构的另一个意外收获是：它让元代谢的行为变得**可审计**。每一层的日志格式不同——层1 记录 `[SHELL] cmd="..." → exit=0`，层2 记录 `[LLM→SHELL] input="..." → parsed="..." → exec_result`，层3 记录 `[CHAT] ...`。出问题时能快速定位是哪个环节。

这本质上是一次"意识架构"的重构——不是给 AI 加更多能力，而是让已有的能力分层执行，用最小的成本解决最常见的问题。对任何在构建自主 Agent 系统的工程师来说，这个模式值得参考。
